## 1. 전체 진단

5단계 반응형 패치 사이클의 한계는 **정적 set 자체**가 아니라 **수동 관리**에 있음. 정적 set 폐지가 아닌 "동적 마스터 + 정적 안전망 합집합" 이 회귀 0 + 운영 안정의 유일한 경로. NULL 결손은 침묵 무시도 보수 BLOCK도 아닌 **UNKNOWN 카테고리화 + 별도 알림** 으로 가시화. 캐시는 발주 윈도우(07:20~08:00) 일관성을 위해 daily_job 1회 빌드가 최적.

## 4. 통합 구현 순서 (15 step)

1. **[Infra]** `infrastructure/database/repos/bundle_stats_repo.py` — `fetch_bundle_stats()` mid별 (bundle_pct, null_ratio, total, unit1)
2. **[Domain]** `domain/order/bundle_classifier.py` — `classify(stats) → {STRONG, WEAK, UNKNOWN}`, 임계값 상수 (70/50/null_30)
3. **[Test]** bundle_classifier 단위 테스트 — 경계값(69.9/70.0/49.9/50.0), NULL 30/31%, total<10
4. **[Application]** `application/order/bundle_master_service.py` — `build_master() → BundleMaster(strong, weak, unknown, snapshot_date)`
5. **[Application]** 정적 fallback 합집합 — `BUNDLE_SUSPECT_MID_CDS ∪ strong ∪ weak`, source_map 기록(static/dynamic/both)
6. **[Cache]** `config/cache/bundle_master_YYYYMMDD.json` 스냅샷 + 메모리 dict
7. **[Constants]** `BUNDLE_SUSPECT_MID_CDS` 정적 set 보존 + 주석 (fallback only)
8. **[Integration]** 발주 파이프라인 기존 참조점을 `BundleMasterService.get()` 호출로 교체. STRONG/WEAK 동일 BLOCK
8.5 **[Policy]** food/dessert mid (001~005, 012)는 BlockClassifier 결과 무관 BLOCK 제외, WARN 로그만 — food-underprediction-secondary 와 조화
9. **[Daily Job]** Phase 0 또는 신규 phase 에 `build_master()` 추가 — 수집 직후, 발주 직전
10. **[Alert]** 전일 스냅샷 diff → 추가/제거 mid 카톡 통지
11. **[Alert]** UNKNOWN(NULL>30%) mid 목록 — "수집 결함 의심" 별도 카톡
12. **[Regression]** 현재 22 BLOCK mid 모두 새 합집합에 포함되는지 회귀 테스트
13. **[Regression]** DB 장애 시뮬레이션(repo 빈 결과) → 정적 fallback만 으로 22 BLOCK 유지 검증
14. **[Observability]** order_logs 메타데이터에 `bundle_block_source` 기록 (schema 변경 없이 dict 로)
15. **[Validation]** 04-09 1차 수정 검증과 병행 — 04-10~04-12 3일 STRONG/WEAK/UNKNOWN 분포 모니터링 → WEAK→STRONG 승격 판단
