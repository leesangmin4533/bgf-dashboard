## Tier 1: 즉시 적용 — 데이터 품질 대폭 향상

### 1. 현재고 조회 (STJK010_M0) — 캡처 완료 (부분)
- **API**: `/stjk010/selLarge` ✅ 캡처됨 (body 639자, resp 598자, ds_ListT1)
- **추가 필요**: `/stjk010/selSearch` (대분류 선택 후 상품별 상세)
- **효과**: BGF 공식 재고 수량 → realtime_inventory 교차 검증
- **현재 문제**: realtime_inventory.stock_qty는 매출 차감 + TTL 기반 추정 → 오차 큼
- **구현 방법**: Direct API로 일 1회 재고 스냅샷 → DB 보정
- **난이도**: ★★☆ (selSearch 추가 캡처 필요)
- **예상 효과**: 재고 정확도 30%+ 향상

### 2. 일자별 재고추이 (STJK030_M0) — 캡처 완료 (부분)
- **API**: `/stjkz10/selLarge` ✅ 캡처됨 (body 598자, resp 486자, dsList)
- **⚠️ 주의**: 화면 ID는 STJK030이지만 API 경로는 `/stjkz10/`
- **추가 필요**: `/stjkz10/selSearch` (대분류 선택 후 상세)
- **효과**: 상품별 재고 변화 패턴 → 회전율 계산 → 발주량 보정
- **현재 문제**: 재고 회전율 데이터 없음 → 안전재고 산정이 매출 기반만
- **구현 방법**: 주 1회 수집 → inventory_trend 테이블
- **난이도**: ★★☆
- **예상 효과**: 과잉재고/결품 15%+ 감소

### 3. 입고예정 내역 조회 (STGJ300_M0) — ✅ 캡처 완료
- **API 1**: `/stgj300/searchAisPlanYmdList` ✅ (body 578자, resp 199자, dsAcpYmd)
  - 응답: `STORE_CD`, `ORD_YMD`, `VIEW_Y...` — 입고 예정 날짜 목록
- **API 2**: `/stgj300/searchChitListPopup` ✅ (body 585자, resp 610자, dsListPopup)
  - 응답: `NAP_PLAN_YMD`(납입예정일), `DGFW_YMD`(도착예정일) — 전표별 입고 상세
- **효과**: 입고 예정일 사전 파악 → 리드타임 정확도 향상
- **현재 문제**: 리드타임을 과거 패턴으로만 추정 → 실제 배송 지연 미반영
- **구현 방법**: 매일 Phase 1.05 삽입 → expected_receiving 테이블
- **난이도**: ★★☆ (2개 API 호출이지만 body 구조 유사)
- **예상 효과**: 리드타임 예측 정확도 40%+ 향상
- **연동**: `receiving-delay-analysis` 기능과 직접 연결

### 4. 시간대별 매출 정보 (STMB010_M0) — ✅ 캡처 완료
- **API**: `/stmb010/selDay` ✅ (body 1116자, resp 1765자, dsList)
  - 응답: `HMS`(시간대), `AMT`(매출액), `CNT`(건수) — 24시간 데이터
- **효과**: 시간대별 판매 패턴 → 1차/2차 배송 수요 분리
- **현재 문제**: `DELIVERY_TIME_DEMAND_RATIO` 고정값(1차=0.70, 2차=1.00)
- **구현 방법**: Direct API로 시간대별 매출 수집 → 동적 비율 계산
- **난이도**: ★☆☆ (body 캡처 완료, SSV 파싱만 필요)
- **예상 효과**: 배송차수별 발주 정확도 20%+ 향상
- **연동**: `FOOD_WEATHER_CROSS_COEFFICIENTS` 시간대 보정에도 활용

### 5. 상품 유통기한 관리 (STCM130_M0) — ✅ 캡처 완료
- **API**: `/stcm130/search` ✅ (body 721자, resp 2027자, dsList)
  - 응답: `LARGE_NM`, `MID_NM`, `EXPIRE_...` — 대/중분류별 유통기한 관리 상품
- **효과**: 공식 유통기한 → TTL 시스템 정확도 향상
- **현재 문제**: `product_details.expiration_days` 일부 데이터만, 나머지는 `FOOD_EXPIRY_FALLBACK` 사용
- **구현 방법**: 주 1회 수집 → product_details.expiration_days 업데이트
- **난이도**: ★☆☆ (body 캡처 완료, 1회 수집으로 충분)
- **예상 효과**: 유령재고 정리 정확도 50%+ 향상
- **연동**: `inventory_repo._get_stale_hours_for_expiry()` TTL 계산 자동화
