# Plan: 이슈 체인 템플릿 설계 (issue-chain-template)

> 작성일: 2026-04-05
> 상태: Plan

## 1. 문제 정의

버그 수정 시 이전 시도의 실패 이유가 git 커밋에 흩어져 있어, 같은 접근을 반복하는 병목이 발생.
실제 사례: 폐기 추적 5단계 패치 누적 (12개 커밋 역추적에 30분+ 소요).

## 2. 목표

- 기능 영역별 "문제 → 시도 → 결과 → 교훈" 체인을 표준화
- 다음 세션의 Claude가 CLAUDE.md 규칙에 의해 강제로 참조/갱신
- 연관 이슈 링크, 검증 체크포인트, 실패 패턴 태그 포함

## 3. 범위

### 포함
- 이슈 체인 문서 표준 템플릿 (`docs/05-issues/_TEMPLATE.md`)
- CLAUDE.md 규칙 갱신 (템플릿 참조 추가)
- 기존 expiry-tracking.md를 템플릿에 맞게 리팩터링

### 제외
- 자동화 도구 (스크립트로 이슈 생성 등) — 수동으로 충분
- 심각도 점수/대시보드 — 이슈가 많지 않아 과함

## 4. 핵심 요구사항

| ID | 요구사항 | 우선순위 |
|----|----------|:---:|
| R1 | 표준 헤더 (상태 태그, 최종 갱신일, 현재 상태) | 필수 |
| R2 | 이슈별 "시도 N" 형식 (커밋해시, 날짜, 결과 ✓/✗) | 필수 |
| R3 | 교훈 섹션 (다음 수정자가 같은 삽질 방지) | 필수 |
| R4 | 연관 이슈 링크 (`→ {파일}#{이슈}`) | 필수 |
| R5 | 검증 체크포인트 (체크리스트, 날짜) | 필수 |
| R6 | 실패 패턴 태그 (`#stale-data`, `#data-format` 등) | 권장 |
| R7 | 영역 분류 기준 | 필수 |

## 5. 영역 분류 기준

| 영역 파일 | 대상 |
|----------|------|
| `expiry-tracking.md` | 폐기 추적, 유통기한 알림, inventory_batches |
| `prediction.md` | 수요 예측, ML 앙상블, 계수 조정 |
| `order-execution.md` | 발주 실행, Direct API, Selenium 저장 |
| `data-collection.md` | 판매/입고/폐기전표 수집, BGF 사이트 접속 |
| `scheduling.md` | 스케줄러, Phase 순서, 타이밍 충돌 |

새 영역이 필요하면 이슈가 3건 이상 쌓일 때 분리.
