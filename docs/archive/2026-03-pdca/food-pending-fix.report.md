## 1. 개요

### 문제
샌드위치(004, 유통기한 2일)와 햄버거(005, 유통기한 3일)가 발주 후 다음날 또 발주되는 중복발주 버그.

### 근본 원인
`FOOD_SAME_DAY_MID_CDS = {'001','002','003','004','005'}` 상수가 **mid_cd 기반**으로 pending을 일괄 무시. "당일배송=당일소진" 논리는 유통기한 1일(001~003)에만 해당하나, 유통기한 2~3일인 004/005도 포함되어 있었음.

### 불일치 구조

| 로직 위치 | 조건 | 004(2일) | 005(3일) |
|-----------|------|:--------:|:--------:|
| 수집기/예측 (pending 무시) | `mid_cd in {001~005}` | pending 무시 ❌ | pending 무시 ❌ |
| 미입고조정 (재고/pending) | `expiration_days <= 1` | 정상 반영 ✅ | 정상 반영 ✅ |

→ 수집기에서 pending을 저장하지 않고, 예측에서도 무시 → 다음날 발주 시 어제 발주분을 모르고 재발주

## 6. 설계 교훈

### 조건 기준 불일치 패턴
동일한 의도("단기유통 푸드 특수처리")에 대해 두 가지 서로 다른 조건이 사용됨:
- **mid_cd 기반**: `{001,002,003,004,005}` — 카테고리 전체 일괄
- **유통기한 기반**: `expiration_days <= 1` — 실제 유통기한 기준

→ **교훈**: 동일 비즈니스 규칙은 동일 조건을 사용해야 함. mid_cd 기반 하드코딩보다 유통기한 기반 조건이 더 정확함.

### 후행 덮어쓰기 방지 (기존 체크포인트 강화)
수집기에서 pending 저장을 스킵하면, 이후 모든 단계에서 pending 데이터가 부재 → 아무리 올바른 pending 차감 로직이 있어도 무의미.

→ **교훈**: 데이터 수집 단계의 필터링은 하류 전체에 영향. 수집 레벨 변경 시 파이프라인 끝까지 추적 필수.
