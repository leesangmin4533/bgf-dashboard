# Design: beverage-decision (음료 발주 유지/정지 판단 시스템)

> 버전 0.2 | 2026-03-05 | BGF리테일 (CU) 자동발주 시스템 연동용

## 1. 프로젝트 개요

CU 편의점 음료 카테고리의 발주 유지/정지 판단 로직을 정의합니다.
디저트 판단 시스템(v1.2)의 아키텍처를 확장하되, 음료 카테고리 고유의 특성을 반영합니다.

### 1.1 디저트 vs 음료 핵심 차이

| 항목 | 디저트 (014) | 음료 (039~048) |
|------|-------------|----------------|
| 상품 수 | 174개 | 972개 (5.6배) |
| 중분류 수 | 1개 | 10개 |
| 소분류 커버리지 | 99.4% | 77.8% (22% 누락) |
| 유통기한 커버리지 | 98.3% | 78% (22% NULL) |
| 전체 폐기율 | ~1% | 0.26% |
| 핵심 리스크 | 폐기 비용 | 매대 비효율 |
| 유통기한 범위 | 2~324일 | 6~9,999일 |
| 계절 변동 | 낮음 | 높음 (얼음 여름 +49%, 원액 +61%) |
| 행사 비율 | 낮음 | 49% (474/972) |

### 1.2 적용 범위

- **대상**: 음료 카테고리 (mid_cd 039~048) 전체 972개 상품
- **판단 방식**: 폐기율 + 매대효율(소분류 중위값 대비 판매 비율) 복합 기준
- **핵심 원칙**: 폐기 리스크가 높은 소수 상품은 엄격하게, 나머지 대다수는 매대효율 기반으로 느슨하게

## 2. 카테고리 분류 체계

### 2.1 분류 로직 (4단계)

소분류 22% + 유통기한 22% 이중 NULL이 존재하므로 중분류(100% 커버)를 1차 기준으로 사용합니다.

```
1차: 중분류코드 기반 (100% 커버)
  - 46 → 카테고리 A
  - 47 → 소분류로 세분화 (흰우유=A, 가공유=B)
  - 42 → 소분류로 세분화 (냉장커피=B, 캔/병커피=C)
  - 39 → 소분류로 세분화 (냉장주스=B, 과즙음료=C)
  - 40, 43, 44, 45 → 카테고리 C
  - 41, 48 → 카테고리 D

2차: 소분류가 있으면 세분화
  - 중분류 내에서 냉장/상온 분리
  - 소분류 없는 상품 → 상품명 키워드로 냉장/상온 추정 (2.4절)

2.5차: 유통기한 NULL 폴백
  - 유통기한이 NULL인 상품 → MID_CD_DEFAULT_EXPIRY 적용 (2.5절)

3차: 안전장치 — 유통기한 불일치 보정
  - C/D로 분류되었는데 유통기한 ≤ 20일 → A로 상향
  - C/D로 분류되었는데 유통기한 21~60일 → B로 상향
  - WARNING 로그 기록

폴백: 중분류코드도 없으면 → C
```

### 2.2 카테고리 정의

| 카테고리 | 설명 | 대상 소분류 | 상품 수 | 유통기한 | 폐기 리스크 | 판단 주기 |
|---------|------|-----------|--------|---------|-----------|---------|
| A | 냉장 단기 유제품 | 호상요구르트(14), 발효유(40), 흰우유(15) | 69개 | 6~30일 | 최대 | 주 1회 |
| B | 냉장 중기 음료 | 가공유(86), 냉장커피(60), 냉장주스(32) | 178개 | 17~168일 | 중간 | 2주 1회 |
| C | 상온 장기 음료 | 탄산(103), 캔/병커피(66), 차(64) 등 13개 소분류 | 479개 | 55~475일 | 낮음 | 월 1회 |
| D | 초장기/비소모품 | 국산생수(19), 수입생수(1), 일반얼음(5), 아이스드링크얼음(5) | 30개 | 120일~inf | 거의 없음 | 월 1회 |
| ? | 소분류 누락 | 중분류 기반 분류 (2.1절) | 216개 | 다양 | 중분류별 | 중분류별 |

> **주**: 상품 수는 소분류 확정분. ?의 216개는 분류 로직에 따라 A~D로 배분됩니다.
> 예: 046 소분류 없는 10개 → A로 배분, 042 소분류 없는 44개 → 대부분 C로 배분.

### 2.4 소분류 없는 상품의 상품명 키워드 분류

중분류 042(커피), 047(우유), 039(주스)는 소분류에 따라 냉장/상온이 달라지므로,
소분류 없을 때 상품명 키워드로 추정합니다.

```python
# 중분류 042 (커피) — 냉장커피 vs 캔/병커피
COLD_COFFEE_KEYWORDS = ['컵', '냉장', '팩', '스트로우']
CAN_COFFEE_KEYWORDS = ['캔', 'CAN']
PET_PATTERN = r'P\d{3}|PET|\d{3,4}ml'
# 판별 순서: COLD→B, CAN→C, PET→C, 폴백→C

# 중분류 047 (우유) — 흰우유 vs 가공유
PLAIN_MILK_KEYWORDS = ['흰우유', '저지방우유', '무지방', '서울우유1L']
# 나머지 → B (가공유). 실측: 소분류 없는 27개 전부 가공유

# 중분류 039 (주스) — 냉장주스 vs 과즙음료
COLD_JUICE_KEYWORDS = ['냉장', '생', '착즙', 'NFC']
# 나머지 → C (과즙음료)
```

### 2.5 유통기한 NULL 폴백

유통기한 NULL 상품(22%)에 중분류별 기본 유통기한을 적용합니다.
3차 안전장치(유통기한 기반 상향)가 정상 작동하도록 보장합니다.

```python
MID_CD_DEFAULT_EXPIRY = {
    '046': 15,    # 요구르트 (실측 9~26일, 보수적)
    '047': 30,    # 우유 (흰우유 44일, 가공유 54일의 하한)
    '042': 90,    # 커피 (냉장 17일 vs 캔 360일, 상온 가정)
    '039': 60,    # 주스 (냉장 24일 vs 상온 180일+)
    '040': 360,   # 건강/에너지
    '043': 365,   # 차
    '044': 365,   # 탄산
    '045': 180,   # 원액
    '041': 365,   # 생수
    '048': 9999,  # 얼음
}
```

### 2.6 판매 지표 계산식

**판매율** (디저트와 동일):
```
판매율 = 판매수량 / (판매수량 + 폐기수량)
분모 0 → 0.0 반환
```

**매대효율지표** (v0.2 — 회전율 대신 도입):
```
매대효율지표 = 해당 상품 주간판매량 / 소분류 중위 주간판매량
- 소분류 내 상품 1개뿐이면 → 중분류 중위값으로 폴백
- 소분류 없는 상품 → 중분류 중위값 사용
- 1.0 = 소분류 평균 수준
```

> 회전율은 재고 데이터 정확도 확보 후 v0.3에서 도입 예정.
> (realtime_inventory에 mid_cd 없음, daily_sales.stock_qty는 스냅샷)

**폐기 손익 계산** (원가 기준):
```
폐기 원가 = disuse_qty × sell_price × (1 - margin_rate / 100)
판매 마진 = sale_qty × sell_price × margin_rate / 100
→ "폐기 원가 > 판매 마진" 이면 순손실 → 즉시 정지 권고
margin_rate NULL → 기본 마진율 60% 적용 (음료 평균 56~67%)
```

## 4. 카테고리별 판단 기준

### 4.0 전 카테고리 공통 — 행사 보호 규칙

음료의 49%(474/972)가 행사 중이므로, 일괄 4주 보호 대신 **행사 유형별 차등** 적용합니다.

```python
PROMO_PROTECTION_WEEKS = {
    '1+1': 3,    # 판매량 2배 급등 → 반동 큼
    '2+1': 2,    # 1+1보다 반동 적음
    '할인': 1,    # 반동 작음
    '증정': 1,
}
# promo_end + PROMO_PROTECTION_WEEKS[promo_type] 이내 → 판단 유예
# promo_type NULL → 보호 비적용 (정상 판단)
# 카테고리 C/D는 폐기 리스크 없으므로 행사 보호 생략 가능 (옵션)
```

이 규칙은 4.1~4.4 모든 카테고리 판단 전에 먼저 체크합니다.

### 4.1 카테고리 A: 냉장 단기 유제품 (주 1회 판단)

폐기 리스크 최대. 디저트 카테고리 A와 유사.

| 생애주기 | 판단 기준 | 결과 |
|---------|---------|------|
| 신상품 (3주 이내) | 무조건 유지. 주간 하락률 50%↑ 시 "급락 경고" 태그 | KEEP / WATCH |
| 성장/하락기 (3~8주) | 주간 판매율 50% 미만 2주 연속 | STOP_RECOMMEND |
| 정착기 (8주+) | 주간 판매량 < 소분류 평균의 30% | STOP_RECOMMEND |
| 전 구간 공통 | 폐기 원가 > 판매 마진 | 즉시 STOP_RECOMMEND |

실데이터: 발효유 0.89%, 호상요구르트 0.54%, 흰우유 0.75%

### 4.2 카테고리 B: 냉장 중기 음료 (2주 1회 판단)

가공유, 냉장커피, 냉장주스. 냉장 공간 제한.

| 생애주기 | 판단 기준 | 결과 |
|---------|---------|------|
| 신상품 (4주 이내) | 보호 기간 — 추세 모니터링만 | KEEP |
| 이후 | 판매율 40% 미만 3주 연속 **또는** 매대효율지표 < 0.20 (소분류 중위값 기준) | STOP_RECOMMEND |

### 4.3 카테고리 C: 상온 장기 음료 (월 1회 판단)

479개 상품, 13개 소분류. 매대 공간 효율이 핵심.

| 생애주기 | 판단 기준 | 결과 |
|---------|---------|------|
| 신상품 (6주 이내) | 보호 기간 — 추세 모니터링만 | KEEP |
| 이후 | 매대효율지표 < 0.15 (**소분류** 중위값 기준) | STOP_RECOMMEND |

> 13개 소분류: 일반탄산(103), 캔/병커피(66), 차(64), 과즙(49), 건강(37),
> 음료원액(33), 에너지(30), 스포츠이온(29), 커피원액(24), 두유(18),
> 숙취해소(16), 논알콜맥주(8), 비타민(2)

**계절 비수기 보정**:

```python
SEASONAL_OFF_PEAK = {
    '048': [11, 12, 1, 2],    # 얼음: 비수기 (실측 -49%)
    '045': [11, 12, 1, 2],    # 원액(아이스드링크): 비수기 (실측 -61%)
}
# 비수기: 매대효율지표 기준을 0.15 → 0.075로 완화
# 이온음료(040 소분류) 등은 1년 데이터 축적 후 추가
```

### 4.4 카테고리 D: 초장기/비소모품 (월 1회 판단)

생수, 얼음. 필수 상비 품목.

| 판단 기준 | 결과 |
|---------|------|
| 월간 판매량 0개 3개월 연속 | STOP_RECOMMEND |

### 4.5 auto_confirm 카테고리별 차등

기존 30일 일괄 → 카테고리별 차등. 판단 주기와 일관성 유지 (각 카테고리 약 2회 판단 후 자동 확정).

```python
AUTO_CONFIRM_DAYS = {
    'A': 14,    # 2주 무판매 → 자동 확정
    'B': 30,    # 1개월
    'C': 60,    # 2개월
    'D': 120,   # 4개월 (필수품이므로 느슨)
}
```

### 4.6 연속 미달 판단 방식

디저트와 동일: 이전 판단 기록 참조 없이, 매번 daily_sales 직접 조회하여 최근 N주 역산.

### 4.7 OrderFilter 연동

디저트와 동일: STOP_RECOMMEND만으로는 발주 차단 안 함.
운영자가 CONFIRMED_STOP으로 확인한 상품만 OrderFilter에서 제외.

## 5. 전체 판단 흐름

```
1. 상품 입력
   → 중분류 1차 + 소분류 2차 + 유통기한NULL폴백 2.5차 + 안전장치 3차
   → 카테고리 A~D 분류

2. 소분류 없는 상품
   → 중분류 기본 카테고리 적용 (042/047/039는 키워드 보조)

3. 첫 입고일 기준 생애주기 판별 (디저트 hotfix 동일)

4. 행사 보호 체크
   → promo_end + 보호기간 이내면 판단 유예 (유형별 차등)

5. 계절 비수기 체크
   → SEASONAL_OFF_PEAK에 해당하면 기준 완화

6. 카테고리별 주기에 맞춰 판매 데이터 집계

7. 판매율 + 매대효율지표 + 폐기 손익 + 추세 계산
   → 카테고리 B/C는 소분류 중위값 기준

8. 유지 / 관찰 / 정지 권고 분류

9. auto_confirm: 카테고리별 차등 무판매 기간 초과 시 자동 확정

10. 사람 최종 확인 후 발주 상태 변경
    → CONFIRMED_STOP만 OrderFilter 반영
```

## 6. 구현 설계

### 6.1 DB 테이블 — 리네이밍 + 통합 (v53)

```sql
-- v53 마이그레이션
ALTER TABLE dessert_decisions RENAME TO category_decisions;

ALTER TABLE category_decisions ADD COLUMN category_type TEXT DEFAULT 'dessert';

UPDATE category_decisions SET category_type = 'dessert' WHERE mid_cd = '014';

CREATE INDEX idx_cat_decisions_type ON category_decisions(category_type, store_id);
```

> **컬럼 dessert_category**: SQLite는 ALTER COLUMN RENAME 미지원.
> 컬럼명은 `dessert_category` 유지하되, 코드에서 `item_category` alias로 사용.
> 향후 테이블 재생성 시 정식 리네이밍.

기존 dessert_decisions 스키마에 추가되는 컬럼:
- `category_type TEXT DEFAULT 'dessert'` — 'dessert' / 'beverage' / 향후 확장

## 7. 예외 처리
